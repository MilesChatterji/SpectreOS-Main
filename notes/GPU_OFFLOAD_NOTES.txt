KERNEL 7 UPDATE (2026-04-22)
-----------------------------
Two issues discovered and resolved after migrating to kernel 7:

1. NVIDIA EGL + VULKAN ICDs HOLDING GPU AWAKE
   On kernel 7, niri was appearing in nvidia-smi at 2MiB (type G) and the GPU was
   sitting at P3/~10W instead of suspending. Root cause: libglvnd loads ALL EGL vendors
   (10_nvidia.json + 50_mesa.json) when niri initializes its GBM/EGL context, causing
   libnvidia-eglcore.so to load and open /dev/nvidia0 + /dev/nvidiactl. VK_ICD_FILENAMES
   alone was not enough — EGL (not just Vulkan) was the remaining path.

   Fix: Added BOTH to the niri-session-amd wrapper (export, systemctl set-environment,
   exec env, and X-KDE-Wayland-Environment desktop entry line):
   - VK_ICD_FILENAMES=/run/opengl-driver/share/vulkan/icd.d/radeon_icd.x86_64.json
   - __EGL_VENDOR_LIBRARY_FILENAMES=/run/opengl-driver/share/glvnd/egl_vendor.d/50_mesa.json

   The nvidia-offload wrapper unsets both so NVIDIA-offloaded apps still get full GPU access.

2. POWER MANAGEMENT WAS ONLY COVERING SYSTEM SLEEP, NOT IDLE SUSPEND
   powerManagement.enable = true only saves/restores GPU state on system suspend/resume.
   It does NOT allow the GPU to power down while the system is running idle (RTD3).

   hardware.nvidia.powerManagement.finegrained = true would fix this, but it requires
   hardware.nvidia.prime.offload.enable = true, which needs explicit PCI bus IDs and
   conflicts with our environment-variable-based offload approach.

   Fix: Manually replicated what finegrained does internally:
   - boot.extraModprobeConfig: options nvidia NVreg_DynamicPowerManagement=0x02
   - services.udev.extraRules: sets power/control=auto for all NVIDIA PCI functions
     (VGA 0x030000, 3D 0x030200, Audio 0x040300, USB 0x0c0330)

   NOTE: RTD3 requires no open handles on the NVIDIA device. The VK_ICD_FILENAMES
   fix above is what makes this effective — without it, niri would still hold the device
   open and block suspend.

SOLUTION IMPLEMENTED
--------------------
✅ The issue was resolved by:

1. **Correcting Device Mapping**: Identified that renderD128 = AMD, renderD129 = NVIDIA
   (Initial assumption was backwards)

2. **Niri Config File**: Added explicit render device configuration in ~/.config/niri/config.kdl:
   ```kdl
   debug {
       render-drm-device "/dev/dri/renderD128"
   }
   ```
   This forces Niri to use the AMD render device instead of auto-detecting.

3. **Environment Variables**: Updated all GBM_BACKEND references from renderD129 to renderD128
   - Updated in niri.nix wrapper script
   - Updated in desktop entry
   - Updated in systemd user environment

4. **NVIDIA Modesetting**: Set `hardware.nvidia.modesetting.enable = false` in gpu-offload.nix
   - Prevents NVIDIA from being initialized for display
   - NVIDIA can still be used for compute/rendering via offload
   - AMD iGPU handles all display output

5. **Verification**: Confirmed via logs and environment checks:
   - Niri logs show: "using as the render node: /dev/dri/renderD128"
   - Environment variables correctly set in Niri and Xwayland processes
   - NVIDIA GPU is idle (0% utilization, minimal power draw)
   - Display is working correctly

HOW TO GENERALIZE FOR DISTRIBUTION
-----------------------------------
For a general distribution, the following needs to be made configurable:

1. GPU Detection:
   - Auto-detect which /dev/dri/card* is iGPU vs dGPU
   - Auto-detect render node numbers (renderD*)
   - Support Intel iGPU (iris/i965) in addition to AMD (radeonsi)
   - Support different NVIDIA GPU models

2. Configuration Options:
   - Make GPU device paths configurable (or auto-detected)
   - Make Mesa driver override configurable based on detected iGPU vendor
   - Allow user to choose default GPU (iGPU vs dGPU)
   - Allow user to enable/disable GPU offload wrappers

3. Files to Generalize:
   - gpu-offload.nix: Make device paths and driver names configurable
   - niri.nix: Make card numbers and render nodes configurable
   - Wrapper scripts: Use detected values instead of hardcoded paths

4. Detection Script Needed:
   - Script to identify iGPU vs dGPU
   - Script to find correct render node for iGPU
   - Script to determine Mesa driver name based on iGPU vendor

5. What Can Be Removed for General Distribution:
   - Hardcoded card numbers (card0, card1)
   - Hardcoded render node numbers (renderD128, renderD129)
   - Hardcoded GPU model names (Radeon 890M, RTX 4050)
   - Hardcoded Mesa driver (radeonsi) - should detect Intel vs AMD
   - Power consumption values specific to this hardware
   - ⚠️ IMPORTANT: Must auto-detect which render node corresponds to which GPU!
     The mapping (renderD128=AMD, renderD129=NVIDIA) is hardware-specific

6. What Should Stay:
   - General GPU offload pattern
   - Environment variable approach
   - Wrapper script concept
   - Power management configuration
   - Module structure (separate files for GPU config)

EXAMPLE GENERALIZED CONFIGURATION
----------------------------------
Instead of:
  WLR_DRM_DEVICES=/dev/dri/card1
  GBM_BACKEND=/dev/dri/renderD128  # ⚠️ Note: renderD128 is AMD, not renderD129!
  MESA_LOADER_DRIVER_OVERRIDE=radeonsi
  Niri config: render-drm-device "/dev/dri/renderD128"

Use:
  WLR_DRM_DEVICES=${config.hardware.gpu.igpu.device}
  GBM_BACKEND=${config.hardware.gpu.igpu.renderNode}  # Must detect AMD render node
  MESA_LOADER_DRIVER_OVERRIDE=${config.hardware.gpu.igpu.mesaDriver}
  Niri config: render-drm-device "${config.hardware.gpu.igpu.renderNode}"

Where these values are auto-detected or user-configured.
⚠️ CRITICAL: The render node detection must correctly identify which render node
belongs to the iGPU, as the numbering (renderD128 vs renderD129) is not consistent
across systems and depends on initialization order.
